# Day 02 — Agent Profiles

## Цель

Создать три специализированных агентных профиля (`.claude/agents/`) для AIChallenge KMP проекта:
1. **Bug Fixer** — находит причину бага, чинит, проверяет регрессии
2. **Researcher** — исследует кодовую базу, выдаёт структурированный ответ
3. **Smart Day Builder** — расширение kmp-day-builder с автоанализом предыдущего дня

---

## Контекст проекта

- **Стек:** Kotlin Multiplatform, Compose Multiplatform, Ktor 3.4.3
- **Архитектура:** MVI + Clean Architecture
- **Существующие агенты:** `mvi-checker`, `kmp-day-builder`, `ktor-feature-builder`
- **Тесты:** Только один `ApplicationTest.kt` (GET /) — большинство эндпоинтов не покрыты
- **Server routes:** calc, dev, files, git, local, mcp, notes, pipeline, rag, review, scheduler, support

---

## Профиль 1: Bug Fixer

### Назначение
Получает описание бага → находит root cause в коде → делает минимальный фикс → проверяет похожие паттерны.

### Инструменты
`Read, Glob, Grep, Edit` — без Bash и Write (без деструктивных операций).

### Тест-баг
**Smart cast на delegated property**: `val state by viewModel.state.collectAsStateWithLifecycle()` — smart cast невозможен на nested свойствах делегированного свойства. Паттерн `state.task!!.field` падает с compile error. Фикс: захватить в локальную переменную `val task = state.task`.

### MUST
- Найти все вхождения проблемного паттерна через Grep
- Прочитать полный контекст файла (не только строку с ошибкой)
- Проверить аналогичные паттерны в других Screen.kt файлах
- Описать root cause до фикса

### MUST NOT
- Рефакторить несвязанный код
- Менять архитектуру
- Запускать bash-команды
- Чинить больше, чем сломано

### Формат ответа
```
## Что нашёл
- root cause + файл:строка

## Что починил
- конкретные правки с diff-фрагментами

## Что проверил
- список файлов проверенных на аналогичный паттерн
- найдено N аналогичных мест / нарушений нет
```

## Профиль 2: Researcher

### Назначение
Получает вопрос по кодовой базе → исследует код → выдаёт структурированный аналитический ответ. Только чтение.

### Инструменты
`Read, Glob, Grep` — только чтение, никаких изменений.

### Тест-вопрос
"Какие эндпоинты не покрыты тестами?" — агент должен найти все Routes.kt файлы, сопоставить с тестами, выдать таблицу покрытия.

### MUST
- Искать файлы через Glob (`**/*Routes.kt`, `**/*Test.kt`)
- Читать найденные файлы для извлечения эндпоинтов / тест-методов
- Строить явные связи между файлами
- Указывать конкретные файлы и строки

### MUST NOT
- Редактировать любые файлы
- Писать код или фиксы
- Предлагать архитектурные изменения (если не спросили)
- Галлюцинировать — только то, что нашёл в коде

### Формат ответа
```
## Вопрос
(перефразировать для ясности)

## Исследованные файлы
- список файлов с кратким описанием

## Связи
- схема зависимостей/потоков

## Выводы
- конкретные факты из кода

## Пробелы
- что не удалось найти / требует уточнения
```

## Профиль 3: Smart Day Builder (усиление)

### Назначение
Расширение `kmp-day-builder`: перед созданием нового дня автоматически изучает предыдущий день, извлекает паттерны Agent/Routes/mechanics и предлагает механику для нового дня.

### Инструменты
`Read, Glob, Grep, Edit, Write, Bash`

### Дополнительные шаги (поверх kmp-day-builder)
1. Читает `.specs/dayNN-1.md` предыдущего дня
2. Читает 3 файла предыдущего дня (Agent, Routes, Screen)
3. Извлекает: тип механики (tool-calling/RAG/streaming/pipeline), эндпоинты, паттерны UI
4. Предлагает 2-3 варианта механики для нового дня (если не указана явно)
5. Только после подтверждения создаёт файлы

## Итерации

### Bug Fix — что пришлось доработать
После первой попытки добавлено:
- Явное требование проверять ВСЕ Screen.kt, а не только сообщённый файл
- Формат "что проверил" — обязателен, не опционален

### Research — что пришлось доработать
После первой попытки добавлено:
- Явный запрет галлюцинировать (раздел MUST NOT)
- Требование указывать файлы:строки, а не только имена пакетов
