## Шаг 1: Изучи кодовую базу ПЕРЕД интервью

Прежде чем задавать вопросы — исследуй существующий код, чтобы вопросы были конкретными и по существу, а не абстрактными.

Выполни следующее (параллельно где возможно):

1. **Проверь ast-index**: `ast-index stats` — если индекс отсутствует, запусти `ast-index rebuild`
2. **Изучи архитектуру**: прочитай `CLAUDE.md` и `.specs/` (если есть) — понять стек, паттерны, ограничения
3. **Найди релевантный код**: через `ast-index search <ключевые слова из запроса>` найди существующие похожие фичи, компоненты, сущности
4. **Изучи смежные файлы**: прочитай 2-4 наиболее релевантных файла (контроллеры, стора, компоненты), чтобы понять существующие паттерны и соглашения
5. **Определи точки интеграции**: что уже есть в коде, что затронет новая фича (БД, API, UI компоненты)

Сформируй для себя краткую внутреннюю картину: что уже сделано, какие паттерны используются, куда вписывается новая фича.

## Шаг 2: Проведи глубокое интервью

Используя знание кодовой базы, проинтервьюируй меня через AskUserQuestion буквально обо всём — но вопросы должны быть:
- **Конкретными**: ссылайся на реальные файлы, компоненты, таблицы из кода ("Хранить в существующей `UserSettingsEntity` или новая таблица?")
- **Неочевидными**: не спрашивай про то, что уже понятно из кода или запроса
- **Архитектурными**: трейдоффы, риски, совместимость с существующим кодом
- **UX/UI**: поведение, краевые случаи, взаимодействие с существующими экранами

Покрой темы: техническая реализация, UI и UX, безопасность, трейдоффы, риски, обратная совместимость.

Погружайся максимально глубоко и продолжай интервью непрерывно, пока оно не будет полностью завершено.

После завершения интервью запиши спецификацию в файл `.specs/<feature-slug>.md`.

После записи спецификации — **немедленно запусти полный workflow по профилю "Бизнес-фича"** из CLAUDE.md, используя спецификацию как исходный запрос пользователя:

2. **Plan** — после сбора результатов консилиума запусти агент `Plan` (model: opus) для формирования плана реализации
3. **Executing** — запусти нужные агенты по стеку (builder-spring-feature / builder-compose-feature / vue-expert) для написания кода
4. **Validation** — спроси пользователя на каких платформах тестируем, сформируй E2E сценарий, запусти проверки
5. **Report** — запусти `general-purpose` (model: haiku) для формирования итогового отчёта в `./swarm-report/<slug>-<date>.md`

Все переходы между стадиями — строго по разрешённым путям из CLAUDE.md. Перед каждой сменой стадии явно указывай текущую и следующую стадию.
